## ** Initialization & Dependency Setup **

Follow these steps in chronological order to provision the runtime environment and resolve missing dependencies.
1. Clone the Project and Set Permissions.
   Place the project files into the target server directory and navigate to the root folder:
   ```Bash
   cd /var/www/musicwave
   ```
   
2. Dependency Resolution (Generating the vendor/ folder)
   The project utilizes Composer to handle external libraries. Installing them locally ensures full execution capabilities while keeping the source repository clean.
   Monolog (monolog/monolog): Handles secure, structured application logging conforming to PSR-3 standards.
   Install monolog for manage the log in PHP in the folder '/var/www/musicwave'
   ```bash
   sudo apt update
   sudo apt install composer
   composer require monolog/monolog
   ```

   PHPMailer (phpmailer/phpmailer): Employed for secure email delivery, eliminating common Email Injection flaws found in PHP's native mail() function.
   Install PHPMailer for secure email handling in the folder '/var/www/musicwave'
   ```bash
   composer require phpmailer/phpmailer
   ```

3. Apache2 Web Server Configuration
   Apache must be configured to point directly to the public/ subdirectory. This locks down core backend logic (config/, includes/) from malicious direct URL access.
   Set public folfer for apache2, default is /var/www/html if you want chage it like me follows this command
   ```Bash
   sudo nano /etc/apache2/sites-available/000-default.conf
   ```
   Change the DocumentRoot directive to match the project's public directory:
   DocumentRoot /var/www/musicwave/public
   Manage the Apache2 service using the following commands:

   Start the server
   ```bash
   sudo systemctl start apache2
   ```
   Verify the operational status and check for configuration syntax errors
   ```bash
   sudo systemctl status apache2
   ```
   Stop the server
   ```bash
   sudo systemctl stop apache2
   ```

   To see error log of apache2:
   ```bash
   sudo tail -n 50 /var/log/apache2/error.log
   ```

## Database Background Maintenance & Event Scheduler

MusicWave relies on a MariaDB Event (`purge_expired_pending_users_and_log`) to automatically perform cyclic cleanup operations every hour, such as purging expired registration rate-limits and unconfirmed pending accounts.

### The Volatility Issue on System Shutdown

By default, MariaDB reads its initialization parameters from static configuration files on startup. Running a dynamic runtime query such as `SET GLOBAL event_scheduler = ON;` only activates the engine inside the current volatile memory session. 
**If the host machine is shut down or rebooted**, the database process terminates, and upon the next system startup, the Event Scheduler will revert back to its default state (`OFF`). 
While the specific automated event remains safely saved inside the database dictionary as `ENABLED`, it will not physically execute because the global engine managing it is resting inactive.


### Troubleshooting and Runtime Verification

To diagnose, check, and manually reactivate the event infrastructure via the Bash CLI, use the following interactive steps:
1. Connect to the database monitor with administrative privileges:
   ```bash
   sudo mariadb -u root
   ```
2. Inspect the operational state of the global event background manager:
   ``` mariaDB bash
   SHOW VARIABLES LIKE 'event_scheduler';
   ```
3. Verify that the maintenance worker routine itself is registered and enabled within your database context:
   ``` mariaDB bash
   SHOW EVENTS FROM music_wave_DB;
   ```
4. Boot the scheduler engine back to an active state for the current runtime session:
   ``` mariaDB bash
   SET GLOBAL event_scheduler = ON;
   ```
   
### Permanent Fix (Persistent Across Machine Reboots)

To ensure that the automated background maintenance tasks resume automatically whenever the machine boots up, the parameter must be written into the daemon's static server layout:
1. Open the MariaDB server architecture configuration profile using a text editor:
   ```bash
   sudo nano /etc/mysql/mariadb.conf.d/50-server.cnf
   ```
   (Note: Depending on your system distribution or if you are using standard MySQL, the file destination path could alternative be /etc/mysql/my.cnf)
2. Locate the core service group block marked as [mysqld] and append the permanent execution flag directly beneath it:
   event_scheduler=ON
3. Save the modified profile configuration layer and restart the database service daemon to initialize the permanent background routine runner:
   ```bash
   sudo systemctl restart mariadb
   ```

## Upgrading Password Hashing to Argon2id (Optional)

By default, MusicWave uses **BCRYPT** (`PASSWORD_BCRYPT`) for password hashing. 
This choice ensures maximum out-of-the-box portability across standard PHP environments while maintaining robust protection against brute-force and Rainbow Table attacks, tailored to the application's specific threat model.

However, if your PHP environment natively supports **Argon2id** (the current industry state-of-the-art algorithm, which provides superior resistance against GPU/ASIC-hardware parallel cracking and side-channel attacks), you can easily upgrade the security layer. 

To check if your PHP server supports Argon2id there are two ways to do this on your local server:
1. (via CLI terminal):
   Open your computer's terminal and type:
   ```bash
   php -r "print_r(password_algos());"
   ```
2. (Using a PHP script):
   Create a temporary file called test.php with this code and type it into your browser:
   ```
   <?php
	if (defined('PASSWORD_ARGON2ID')) {
		echo "Argon2id è supportato!";
	} else {
		echo "Argon2id NON è supportato su questo server.";
	}
	?>
   ```

To perform the upgrade, locate the registration and password reset controllers (`register.php` and `reset_password.php`) and update the hashing function call as follows:

```php
// Replace PASSWORD_BCRYPT with PASSWORD_ARGON2ID
$hashed_password = password_hash($password_raw, PASSWORD_ARGON2ID);
```

Note: No changes are required in login.php, as PHP's native password_verify() function automatically detects the algorithm type from the stored database hash prefix.
